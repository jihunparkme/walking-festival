- 현재 선택한 탭 같은 UI 상태
- 임시 안내 문구 닫기 여부

- 서버에 이미 저장된 데이터를 화면에서 빨리 보여주기 위한 캐시
- 시뮬레이션 on/off 같은 테스트용 상태

이 데이터들의 공통점은 삭제되거나 초기화돼도 불편은 생기지만 운영 사고로 이어지지 않는다는 점이다.

## 반드시 서버로 옮겨야 하는 것

- 추첨용 고유 번호
- 참여 여부
- 도장 완료 여부
- 최종 걸음 수 또는 인증 완료 상태

이 데이터들은 지워지면 안 되고, 중복이나 조작도 막아야 하며, 관리자 기준의 단일 진실 원본이 필요하다.

## 이 프로젝트의 최소 권장 구조

1. 서버 DB에 participants 테이블 생성
2. 캠페인 참여 버튼 클릭 시 서버에서 고유 번호 발급
3. 도장 완료 여부와 인증 완료 여부를 서버에 저장
4. 사진은 Vercel 자체가 아니라 Blob, S3, Supabase Storage 등에 저장
5. 브라우저 localStorage 에는 participantId 같은 식별자나 UI 캐시만 저장

## 현실적인 우선순위

- localStorage 유지: 탭 상태, 임시 캐시
- 서버 이전 1순위: 고유 번호, 참여 여부
- 서버 이전 2순위: 도장 상태, 인증 결과
- 스토리지 이전 필수: 인증 사진

## 핵심 원칙

행사 결과에 영향을 주는 데이터는 서버에 저장하고, 화면 편의 데이터만 localStorage 에 둔다.

## 로그인 필요성

- 사용자별 추첨 번호와 걷기 기록을 안정적으로 관리하려면 로그인 또는 그에 준하는 사용자 식별 방식이 있는 편이 좋다.
- localStorage 만으로는 브라우저 변경, 데이터 삭제, 시크릿 모드, 기기 변경 상황에서 동일 사용자를 다시 식별하기 어렵다.
- 추첨 번호 중복 방지, 기록 복구, 관리자 집계 정확성을 위해 서버 기준 사용자 식별 체계가 필요하다.

## 인증 방식 선택지

### 1. 정식 로그인

- 카카오, 네이버, 구글, 휴대폰 인증 등을 통한 로그인
- 장점: 사용자별 기록 관리와 중복 참여 통제에 유리
- 단점: 구현과 운영 부담이 가장 큼

### 2. 간편 인증 기반 식별

- 전화번호 + 인증번호
- 행사 접수번호 + 이름
- 문자 인증 기반 1회 로그인
- 장점: 회원가입보다 가볍고 사용자 식별 신뢰도도 높음
- 단점: 문자 인증 비용 또는 운영 절차가 필요함

### 3. 비로그인 익명 식별

- 서버가 participantId 를 발급하고 브라우저 쿠키 또는 localStorage 에 저장
- 장점: 구현이 가장 간단함
- 단점: 기기 변경, 데이터 삭제, 중복 참여 통제가 약함

## 추천 인증 전략

- 단순 체험용 이벤트: 익명 participantId 도 가능
- 실제 추첨과 경품 지급 포함: 전화번호 인증 또는 소셜 로그인 권장
- 중복 참여 방지와 사후 당첨 확인 필요: 로그인 또는 휴대폰 인증 사실상 필요
- 이 프로젝트에는 정식 회원가입보다는 간편 인증 방식이 더 현실적임

## Vercel 기준 가장 가벼운 인증 구조

- 프론트: 현재 Vite 앱 유지
- 인증 API: Vercel Functions
- DB: Supabase Postgres
- 세션: httpOnly 쿠키
- 문자 인증 사용 시 SMS 제공업체 연동

이 구성이 가벼운 이유는 별도 서버를 운영하지 않아도 되고, 서버리스 함수에서 인증과 외부 API 호출을 처리할 수 있기 때문이다.

## 카카오 로그인 활용안

- 장점: 국내 사용자에게 익숙하고 문자 발송 비용이 없음
- 장점: 사용자별 식별자를 안정적으로 확보 가능
- 장점: 추첨 번호, 도장 상태, 걷기 기록을 사용자 계정에 연결하기 쉬움
- 한계: 실명 1인 1참여를 완전히 보장하지는 못함
- 한계: 카카오 계정이 없는 사용자를 배제할 수 있음

추천 적용 방식:

- 카카오 로그인 후 서버에서 사용자 생성
- 사용자 생성 시 추첨용 고유 번호 1회 발급
- 걷기 기록, 도장 상태, 인증 사진을 사용자 계정에 저장

## 카카오 미사용자 대응 방안

- 기본 인증: 카카오 로그인
- 대체 인증: 휴대폰 번호 OTP
- 권장 운영 방식: 카카오 로그인과 휴대폰 인증을 병행

이 방식은 카카오 사용자는 빠르게 참여시키고, 카카오 미사용자는 문자 인증으로 포용할 수 있어서 행사 접근성과 운영 신뢰성을 함께 확보하기 좋다.

## 카카오 로그인 + Vercel Functions 구현 순서

1. 카카오 디벨로퍼스에서 앱 생성
2. 웹 도메인과 Redirect URI 등록
3. Vercel 환경변수에 카카오 키, 세션 시크릿, DB 접속 정보 저장
4. users, sessions 등 최소 DB 스키마 생성
5. GET /api/auth/kakao/login 구현
6. GET /api/auth/kakao/callback 구현
7. GET /api/me 구현
8. POST /api/auth/logout 구현
9. 로그인 성공 시 서버에서 사용자 생성 또는 조회
10. 신규 사용자면 추첨용 고유 번호 1회 발급
11. 프론트에서 로그인 상태에 따라 참여 흐름 연결
12. 기존 localStorage 기반 운영 데이터를 서버 저장으로 순차 전환

## 보안 원칙

- 외부 API 비밀키는 프론트에 두지 않고 Vercel Functions 환경변수로 관리
- 브라우저는 내 API 만 호출하고, 실제 외부 API 호출은 서버 함수에서 수행
- 고유 번호 발급, 인증, DB 쓰기 작업은 반드시 서버 경유
- localStorage 는 UI 캐시나 임시 상태 정도에만 사용
