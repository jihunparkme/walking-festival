## localStorage 에 둬도 되는 것

- 현재 선택한 탭 같은 UI 상태
- 임시 안내 문구 닫기 여부
- 사용자가 다시 열었을 때 편하게 이어보기 위한 비중요 상태
- 서버에 이미 저장된 데이터를 화면에서 빨리 보여주기 위한 캐시
- 시뮬레이션 on/off 같은 테스트용 상태

이 데이터들의 공통점은 삭제되거나 초기화돼도 불편은 생기지만 운영 사고로 이어지지 않는다는 점이다.

## 반드시 서버로 옮겨야 하는 것

- 추첨용 고유 번호
- 참여 여부
- 도장 완료 여부
- 최종 걸음 수 또는 인증 완료 상태
- 업로드한 인증 사진
- 추첨 대상자 명단
- 관리자 검수 결과나 운영 로그

이 데이터들은 지워지면 안 되고, 중복이나 조작도 막아야 하며, 관리자 기준의 단일 진실 원본이 필요하다.

## 현재 앱 항목별 권장안

- entryNumber: 반드시 서버
- stamps: 가능하면 서버
- steps: 원본 전체 누적값을 신뢰해야 하면 서버
- photoDataUrl: 반드시 서버 스토리지
- tab 같은 UI 상태: localStorage 가능

## 서버 필수 기준

- 번호 발급이 중복되면 안 되는 데이터
- 행사 결과, 당첨, 인증 판단에 직접 쓰이는 데이터
- 운영자가 나중에 조회하거나 집계해야 하는 데이터
- 사용자가 지우거나 조작하면 안 되는 데이터

## 클라이언트 저장 가능 기준

- 화면 편의성
- 임시 입력값
- 재방문 UX 개선용 캐시
- 서버 원본이 따로 있는 데이터의 복사본

## 이 프로젝트의 최소 권장 구조

1. 서버 DB에 participants 테이블 생성
2. 캠페인 참여 버튼 클릭 시 서버에서 고유 번호 발급
3. 도장 완료 여부와 인증 완료 여부를 서버에 저장
4. 사진은 Vercel 자체가 아니라 Blob, S3, Supabase Storage 등에 저장
5. 브라우저 localStorage 에는 participantId 같은 식별자나 UI 캐시만 저장

## 현실적인 우선순위

- localStorage 유지: 탭 상태, 임시 캐시
- 서버 이전 1순위: 고유 번호, 참여 여부
- 서버 이전 2순위: 도장 상태, 인증 결과
- 스토리지 이전 필수: 인증 사진

## 핵심 원칙

행사 결과에 영향을 주는 데이터는 서버에 저장하고, 화면 편의 데이터만 localStorage 에 둔다.
