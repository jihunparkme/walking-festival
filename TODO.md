## 🛠 Tech Stack
* **Frontend & Hosting:** `Vercel Hobby Plan` (Serverless Functions 포함)

## 📌 주요 기능 및 프로세스 흐름

### 1. 사용자 접속 및 등록 (URL 접속)

사용자가 시스템에 최초 접속하여 정보를 입력하고 식별 코드를 발급받는 단계

* **입력 정보:** 이름, 전화번호
* **서버 로직 (전화번호 기준 중복 체크):**
  * **신규 사용자:** 이름, 전화번호, UUID 로그인용 토큰(중복이 안되게 랜덤으로 생성), 추첨번호(중복이 안되게 랜덤으로 생성) DB 저장
  * **기존 사용자:** 이름, 전화번호가 일치하는 데이터가 있을 경우 기존에 발급된 로그인용 토큰 반환 (재접속 처리)
* **클라이언트 저장:** 발급받은 로그인용 토큰을를 `LocalStorage` 및 `Cookie`에 저장하여 이후 모든 API 요청 시 인증 기준으로 활용

> ⚖️ **개인정보 보호 주의사항 (법적 필수 요건)**
> * 전화번호는 개인정보보호법 상 민감한 개인정보에 해당하므로 아래 사항을 반드시 준수합니다.
> * 정보 입력 화면 내 **'개인정보 수집 및 이용 동의'** 문구 필수 노출
> * 이벤트 종료 후 개인정보 **파기 일정 및 프로세스** 명시
> * 참가 시작 직전 오입력 방지를 위한 **'이름 및 전화번호 재확인 얼럿(Alert)'** 팝업 노출

### 2. 스탬프 랠리 (도장판 기능)

각 부스에 부착된 고정 QR 코드를 스캔하여 스탬프를 적립하는 흐름입니다.
* **오프라인 준비:** 부스마다 고유한 `부스ID`가 포함된 고정 QR 코드 부착

#### [참가자 QR 스캔 시 상세 흐름 (서버리스)]
1. 참가자가 카메라로 부스의 QR 코드를 스캔합니다.
2. 스캔 페이지(`{domain}/stamp?booth=booth_01`)로 접속합니다.
3. 스크립트가 `LocalStorage`에서 참가자 `token`을 읽어옵니다.
4. Vercel Serverless Function으로 API 요청을 보냅니다.
    * **Endpoint:** `POST /api/stamp`
    * **Payload:** ```json
      {
      "token": "참가자_고유_토큰",
      "boothId": "booth_01"
      }
      ```
5. **서버(Vercel Function) 처리:**
    * Supabase stamp_records 테이블을 조회하여 해당 부스의 중복 도장 여부를 체크합니다.
    * **신규 도장:** Supabase stamp_records 테이블에 기록을 `INSERT`합니다.
    * **중복 도장:** HTTP 상태 코드 `409 (Conflict)`를 반환합니다.
6. 응답에 따라 프론트엔드 도장판 UI를 즉시 업데이트하여 해당 부스 도장을 표시합니다.
7. 도장판 페이지 접속 시 완료한 부스 정보를 조회 후 각 부스의 id 와 비교해서 도장여부를 표시합니다.

---

### 3. 완주 및 사진 등록
모든 스탬프를 모은 참가자가 완주 QR 코드를 스캔하여 최종 인증 사진을 등록하는 단계입니다. 서버의 부하를 최소화하기 위해 **프론트엔드 직접 업로드(Direct Upload)** 방식을 채택했습니다.

1. 참가자가 완주 QR 코드를 스캔합니다.
2. 디바이스 내에서 사진을 촬영하거나 갤러리에서 선택합니다.
3. 프론트엔드에서 Cloudinary API를 이용해 이미지를 직접 업로드합니다. (**백엔드 서버 필요 없음**)
4. 업로드 완료 후 Cloudinary로부터 최적화된 이미지 URL을 반환받습니다.
5. 반환된 `이미지 URL`만 Supabase의 참가자 레코드에 추가로 저장합니다.
